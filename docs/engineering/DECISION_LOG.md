## Yahoo Finance Price Lag ‚Äî Thai SET Stocks

**Date:** 2026-05 (discovered during snapshot integrity work)  
**Problem:** Yahoo Finance has ~15‚Äì20 minute delay for `.BK` symbols after SET closes at 16:30 ICT. Prices pulled at 16:35 may still reflect the last automated auction price, not final ATC.  
**Decision:** APScheduler `daily_portfolio_snapshot` fires at **17:45 ICT** (Mon‚ÄìFri), giving yfinance time to publish settled ATC prices.  
**Reasoning:** `KBANK.BK` pulled at 16:35 showed 196.5; ATC settled at 197.0 ‚Äî 0.25% error compounding across a portfolio.  
**Impact:** Snapshot timing must not be moved earlier. Applies to automated job, manual Analyze All, optimizer runs, and backfill scripts.  
**Long-term backlog:** Replace yfinance `.BK` close prices with a direct SET data feed guaranteeing ATC settlement.

## Fee Decomposition (Phase 3B.10 + hardening)

**Date:** 2026-05-27  
**Problem:** Brokerage fees were stored as a single total in `Transaction.fees` with `taxes=0`. No visibility into components; DR instruments could not have different structures.  
**Decision:** Decompose into `fees = pre-VAT subtotal` (commission + trading + clearing) and `taxes = VAT`. Total is identical (`0.157% √ó 1.07`), so cash balances don't change.  
**Reasoning:** Decomposition enables per-component transparency, configurable fee profiles per symbol class, and correct DR handling without rewriting accounting logic.  
**Impact:** Old rows have `taxes=0` so `fees + taxes = fees` (unchanged). New rows sum to same total. `period_fees_paid` formula uses `(tx.fees or 0) + (tx.taxes or 0)` for backward compat.

## Fee-Inclusive Cost Basis

**Date:** 2026-05-27 (hardening sprint)  
**Problem:** `avg_cost` was computed as `price_per_share` only, excluding brokerage fees. This understated cost basis and overstated realized P/L.  
**Decision:** `avg_cost = net_buy_amount / shares` where `net_buy_amount = gross + all_fees_incl_vat`. Weighted average on additions uses `net_buy_amount`, not raw price.  
**Reasoning:** Institutional-grade accounting convention. Fees are a real cost of acquiring the position; excluding them understates true acquisition cost.  
**Impact:** `avg_cost` values increase by ~0.168%. Historical P/L slightly lower. `POST /admin/recalculate-cost-basis` endpoint repairs historical data.

## Position Import Accounting Fix (Phase 3B.9)

**Date:** 2026-05-25  
**Problem:** `INITIAL_POSITION` transactions injected equity into the portfolio without a cash outflow. The snapshot engine treated the full market value as investment gain, distorting return %.  
**Decision:** Classify `INITIAL_POSITION` as a NON-PERFORMANCE CAPITAL INFLOW. Store `imported_asset_value` at current market price in the snapshot. Subtract from the performance formula.  
**Reasoning:** A user who imports an existing holding they've had for years should not see a 100% return on the import date. The value existed before the portfolio was tracked.  
**Impact:** `add_holding` endpoint now calls `execute_initial_position()` to create a transaction trace. All return, alpha, Sharpe, and attribution calculations exclude imported assets.

## Backdated Import Detection (Phase 3B.9 Hotfix)

**Date:** 2026-05-25  
**Problem:** The import-stripping logic used `Transaction.transaction_date` to find INITIAL_POSITION events in the snapshot window. Backdated imports (user supplies original purchase date years ago) escaped the filter.  
**Decision:** All non-performance transaction queries filter by `Transaction.created_at` (physical DB insert time), not `transaction_date`.  
**Reasoning:** `created_at` is always `datetime.utcnow()` at insert time regardless of user-supplied `transaction_date`. It is the authoritative indicator of "when this event was recorded in the system."  
**Impact:** Applies to all queries in `portfolio_snapshots.py`. `transaction_date` remains accurate as the user's reported trade date.

## DR Symbol Handling

**Date:** 2026-05 (early architecture)  
**Problem:** DR stocks (e.g. `NVDA01.BK`) are Thai Depository Receipts trading on SET, but yfinance only knows the base ticker (`NVDA`).  
**Decision:** `normalize_dr_symbol()` strips the two-digit suffix for yfinance calls. Original symbol kept in DB for display. Pattern `^[A-Z]+\d{2}\.BK$` identifies DRs.  
**Reasoning:** Keeps all Thai SET symbols consistent with `.BK` suffix for UI while enabling correct yfinance data retrieval.  
**Impact:** Must call `normalize_dr_symbol(symbol)` before any `ticker.info` or `fetch_info` call in agents. Sector resolution for DRs uses FA cache (base ticker data).

## Authorized Exception Semantics (Phase 3B.6)

**Date:** 2026-05-25  
**Problem:** L2/L3 AI agents and the Consensus Engine were treating "Turnover Relaxation Active" as a failure condition and escalating to REVIEW even when the relaxation was intentional and authorized.  
**Decision:** Inject explicit "AUTHORIZED EXCEPTION SEMANTICS" into `_t1_note` (shared across all 3 layer prompts) classifying authorized states vs dangerous failure states. Add governance penalty exemption for turnover flags when Tier 1 relaxation is active.  
**Reasoning:** AI agents lack the context to distinguish an intentional policy relaxation from a genuine violation. The classification must be injected explicitly.  
**Impact:** `TURNOVER_BREACH` in UI displays as blue/informational rather than red/error. Governance score not penalized for authorized expansions.

## Constraint Resolution Hierarchy (Phase 3B.5)

**Date:** 2026-05-25  
**Problem:** Multiple overlapping constraint sources (user settings, regime policy, emergency limits, system safety) could contradict each other with no deterministic resolution.  
**Decision:** 4-source merge with `effective = min(A, B, C, D)` for upper bounds. Resolver runs before policy engine; policy engine applies confidence discount on top.  
**Reasoning:** Clean separation: resolver produces "pre-discount" constraints; policy engine handles dynamic adjustment. Audit trail per-constraint shows binding source.  
**Impact:** `emergency_limit` stored as `float | None` (not `float("inf")`) for JSON serializability. `_norm_sector()` duplicated in `policy_engine.py` to avoid circular import.

## Policy Engine Confidence Discount (Phase 3B.4)

**Date:** 2026-05-24  
**Problem:** The optimizer was equally aggressive regardless of regime confidence or consensus strength.  
**Decision:** Dynamic confidence discount (0‚Äì1) based on regime confidence (50%), stability (35%), consensus strength (15%). Discount tightens constraints proportionally.  
**Reasoning:** Low-confidence regime readings should yield more conservative portfolios. The discount gives a continuous rather than binary risk adjustment.  
**Impact:** At low confidence: cash floor +8%, max position ‚àí15%, turnover ceiling reduced, aggressiveness reduced.

## Persona Per-Portfolio Design (Phase 3B.2)

**Date:** 2026-05-23  
**Problem:** A user may want different investment philosophies for different portfolios (e.g. aggressive growth in one, dividend income in another).  
**Decision:** `Portfolio.strategy_persona` column (TEXT, default BALANCED) ‚Äî one persona per portfolio, not workspace-level.  
**Reasoning:** Per-portfolio is more flexible and requires no breaking schema change for existing portfolios (default BALANCED is neutral).  
**Impact:** DNA computation lightweight ‚Äî uses `ta_score`, `fa_score`, `pe_ratio`, `roe`, `revenue_growth` already in `scores_map`; no extra yfinance calls.

## Shadow Portfolio Design (Phase 3B.7A)

**Date:** 2026-05-25  
**Problem:** No way to measure whether following AI recommendations actually outperforms the user's actual decisions.  
**Decision:** Two shadow types: `STATIC_FROZEN` (frozen at decision time, tracks "what would have happened") and `ACTIVE_MODEL` (refreshed each run, hypothetical 100%-compliant portfolio). Only one ACTIVE_MODEL shadow per portfolio.  
**Reasoning:** STATIC_FROZEN answers "did that recommendation have merit?"; ACTIVE_MODEL answers "what would a fully AI-compliant portfolio look like today?".  
**Impact:** Attribution and calibration depend on accumulated history. BHB sector decomposition is a structural stub awaiting per-sector benchmark data.

## Concurrent Watchlist Analysis

**Date:** 2026-05 (optimization sprint)  
**Problem:** Sequential analysis with sleep delays: ~4 minutes for 68 stocks.  
**Decision:** `asyncio.gather()` with `Semaphore(10)` concurrency cap and 10s per-stock timeout.  
**Reasoning:** AI provider rate limits require concurrency cap; 10s timeout prevents one slow AI call from blocking the entire batch.  
**Impact:** ~32 seconds for 68 stocks. Timed-out stocks fall back to deterministic signal; `AnalysisCache` not written for fallbacks (retry on next run).

## Portfolio Metrics Engine Consolidation

**Date:** 2026-06-30  
**Problem:** `portfolio_rebuilder.py`, `portfolio_snapshots.py`, and `snapshot_return_recovery.py` each independently implemented the same nine period-return fields, with subtle, undocumented divergences (window field, cash-flow formula, signed-vs-absolute quantity correction, duplicate-import handling). `docs/PORTFOLIO_CALCULATION_RULES.md` documented these divergences and recommended canonical answers but made no code changes.  
**Decision:** Extract one pure function, `services/portfolio_metrics.py::compute_period_metrics()`, and migrate all three engines (plus `snapshot_repair.py`, which inherits the fix via delegation) to call it instead of duplicating the formulas. ADR-001 (Transaction ledger is the single source of truth), ADR-002 (Metrics never compensate for ledger corruption), ADR-003 (`transaction_date` governs replay-order; `created_at` governs window-membership in incrementally-built engines only ‚Äî see below), ADR-004 (exactly one implementation) were ratified to guide this.  
**Reasoning:** ADR-004 directly states the goal ‚Äî one formula, three engines, identical output for any given ledger and window. Per-engine duplication was the root cause of every divergence found during the architecture review.  
**Impact:** `portfolio_rebuilder.py` and `portfolio_snapshots.py` were mechanically migrated with zero behavior change (verified by their full existing test suites passing unchanged). `snapshot_return_recovery.py`'s migration intentionally changed two formulas ‚Äî see the next two entries. New regression suite: `test_portfolio_metrics.py` (pure unit tests) and `test_portfolio_metrics_parity.py` (cross-engine parity + the ADR-003 backdated-transaction case).

## External Cash Flow Formula Confirmed Canonical (Implementation A)

**Date:** 2026-06-30  
**Problem:** `portfolio_rebuilder.py`/`portfolio_snapshots.py` summed ledger DEPOSIT/WITHDRAW/INITIAL_CASH events directly ("Implementation A"); `snapshot_return_recovery.py` instead derived the figure from the change in `Portfolio.cash_balance` minus BUY/SELL cash movement ("Implementation B"). `PORTFOLIO_CALCULATION_RULES.md` Section 4 originally recommended B as "self-validating against the authority column."  
**Decision:** Implementation A (ledger-derived) is canonical, reversing the original recommendation.  
**Reasoning:** A follow-up architecture review found B's "self-validation" property inverts Design Principle 1 ‚Äî it treats `Portfolio.cash_balance` (a derived, disposable column per the ledger-is-truth principle) as authoritative input to the return formula, rather than as something checked against the ledger (which is exactly what `ledger_validator.py` CHECK 8, `CASH_MISMATCH`, already does). Worked examples showed A fails loud on `cash_balance` drift (an obviously-wrong return number, easy to catch) while B fails quiet (silently absorbs the drift into a plausible-looking "clean" return, hiding the underlying bug). A real production case ‚Äî phantom DEPOSIT + duplicate INITIAL_POSITION on a specific portfolio/date ‚Äî confirmed this concretely and became the regression test for the fix.  
**Impact:** `snapshot_return_recovery.py` migrated from B to A. Only affects portfolios whose ledger and `cash_balance` have ever drifted apart; forward-only (no backfill of already-persisted snapshots). Test: `test_phantom_deposit_and_initial_position_surface_as_anomalous_return` in `test_snapshot_return_recovery.py`.

## Signed Manual Adjustment Value Fix

**Date:** 2026-06-30  
**Problem:** `manual_adjustment_value` (the QUANTITY_CORRECTION strip in the return formula) used `abs(qty_correction_delta)` in `portfolio_rebuilder.py` and `portfolio_snapshots.py`, but the correct, signed formula in `snapshot_return_recovery.py`. A downward correction therefore double-subtracted in two of the three engines: once for the real NAV drop, once again for a wrongly-positive strip in the same direction, fabricating a loss of roughly 2√ó the correction's market value.  
**Decision:** `manual_adjustment_value = qty_correction_delta √ó price` (signed), matching `snapshot_return_recovery.py`'s original formula, in the shared `compute_period_metrics()`.  
**Reasoning:** This is a correctness bug, not a policy choice ‚Äî an unsigned strip cannot make a pure data-entry correction net to zero effect on return, which Design Principle 2/3 require.  
**Impact:** Changes historical numbers only for a portfolio with a downward `QUANTITY_CORRECTION` that is rebuilt/recomputed after this fix; forward-only, no backfill. Test: `test_quantity_correction_downward_strips_negative_amount` in `test_portfolio_metrics.py`.

## INITIAL_POSITION Dedup Removed from Snapshot Engines

**Date:** 2026-06-30  
**Problem:** `snapshot_return_recovery.py` skipped an `INITIAL_POSITION` transaction's contribution to `imported_asset_value` when the symbol already appeared in the previous snapshot's holdings with equal-or-more shares (a defensive heuristic against duplicate imports). `portfolio_rebuilder.py` and `portfolio_snapshots.py` had no such dedup, creating a three-way inconsistency.  
**Decision:** Remove the dedup heuristic. `compute_period_metrics()` strips every `INITIAL_POSITION` transaction in the window at face value, unconditionally.  
**Reasoning:** Duplicate-import detection and correction is a ledger-quality concern with a dedicated owner ‚Äî `ledger_validator.py` (detection) and `ledger_repair_plan.py` (Phase 6.7E, already shipped, auto-detects and repairs `DUP_INITIAL_POSITION`). Per ADR-002, the metrics layer should not silently compensate for ledger defects that a purpose-built repair pipeline already exists to fix upstream.  
**Impact:** A portfolio with an un-repaired duplicate `INITIAL_POSITION` will show an inflated `imported_asset_value` (and distorted `investment_return_pct`) after this change, where previously the dedup heuristic masked it. Run `validate_ledger` / `generate_repair_plan` on any portfolio that predates this change before trusting recomputed numbers.

## Time Attribution Scope (ADR-003 Resolution)

**Date:** 2026-06-30  
**Problem:** ADR-003 states "`transaction_date` governs investment performance attribution; `created_at` must never affect it." Taken literally, this would require switching `portfolio_snapshots.py` and `snapshot_return_recovery.py`'s window-membership logic from `created_at` to `transaction_date` ‚Äî but `created_at`-windowing in those two engines is the documented fix for the Phase 3B.9 "Backdated Import Detection" bug (a backdated transaction would become invisible to every window check under `transaction_date`-only attribution in an incrementally-built snapshot series, leaking its value into NAV as an undetected phantom gain).  
**Decision:** ADR-003 is scoped to portfolio-state/replay-order questions (cost basis, holdings-as-of-date, chronological sort) ‚Äî the only thing `compute_period_metrics()` itself ever uses a date for (it reads neither field; callers pre-filter). Window-membership logic is unchanged and remains each engine's own responsibility: `created_at` for the two incrementally-built engines, `transaction_date` for the full rebuild.  
**Reasoning:** Applying ADR-003 literally to window membership would reintroduce a previously-fixed, documented production bug, directly conflicting with this refactor's "business behaviour must remain unchanged" constraint. This decision was confirmed explicitly with the task requester before implementation began.  
**Impact:** No behavior change to window membership in any engine. Regression test: `test_backdated_transaction_attributed_per_engine_own_window_field` in `test_portfolio_metrics_parity.py`, demonstrating each engine correctly uses its own window field for the same backdated transaction.

## BH/CENTEL Funding Confusion ‚Äî Optimizer Philosophy Introduced

**Date:** 2026-07-06
**Problem:** With 120k cash on hand, the optimizer recommended BUY CENTEL 30k and REDUCE BH 95k, leaving 186k cash idle. Every number was arithmetically correct, but the plan was not: BH was sold without necessity, fees were paid without purpose, and there was no answer available to "why did it sell BH?" Root cause was architectural, not a math bug ‚Äî the system had no explicit distinction between a **Belief** ("BH should eventually be a smaller share of this portfolio") and an **Execution** decision ("this trade is worth making today"), so a slow-moving allocation preference was rendered as an urgent, unexplained trade. The optimizer also had no concept of **Execution Role** (e.g. Funding Source) separate from a trade's **Reason** (e.g. Portfolio Improvement), so a sale justified on its own merits was misread as cash-raising for a purchase that existing cash already covered three times over.
**Decision:** Adopt `docs/OPTIMIZER_PHILOSOPHY.md` as the constitutional design document for the optimizer, execution planning, and AI evaluation layers. It establishes the hierarchical objective (Capital Preservation ‚Üí Policy Compliance ‚Üí Recommendation Integrity ‚Üí Execution Practicality ‚Üí Expected Improvement ‚Üí Turnover Efficiency ‚Üí Idle-Cash Efficiency), the Belief/Execution Plan separation, the Reason/Execution Role distinction for every trade, and the "guilty until proven necessary" default for trading activity.
**Reasoning:** A weighted-score optimizer can produce this exact failure and still report a good score, because expected improvement and unexplainable turnover are fungible under a price list. A lexicographic hierarchy cannot make this mistake: the BH sale served no tier (not preservation, not compliance, not required for practicality) and its improvement value was marginal, so it should never have shipped. Writing this down as philosophy ‚Äî not just fixing the one incident ‚Äî was necessary because the underlying gap (conflating Belief with Execution, and Reason with Role) could otherwise resurface anywhere a future feature touches trade selection or funding logic.
**Impact:** `docs/OPTIMIZER_PHILOSOPHY.md` is now required reading before modifying optimizer, execution, policy, or AI evaluation logic (see `CLAUDE.md`). `services/execution_plan.py` and `services/funding_source_analysis.py` (Phase UX.2E/UX.2L) already implement the Reason/Execution Role split described here ‚Äî this decision formalizes the philosophy retroactively for the design they follow, and binds all future optimizer work to it going forward.

## Execution Optimization Layer (¬ß7 implemented)

**Date:** 2026-07-06
**Problem:** ¬ß7 of `OPTIMIZER_PHILOSOPHY.md` described the Execution Optimization stage as the fix for the BH/CENTEL incident, but no code implemented it: both execution-plan surfaces ‚Äî the optimizer page's `ExecutionPlanCard` (via `frontend/lib/executionPlan.ts::deriveExecutionPlan`) and the Decision Workspace's `build_execution_plan()` ‚Äî summed every SELL/REDUCE-flagged holding's full release into "cash released" unconditionally, regardless of whether a buy actually needed the cash. The exact founding-example arithmetic (120k cash, 30k need, 95k REDUCE ships anyway, 186k idle) was live in both surfaces.
**Decision:** Added `backend/services/optimizer/execution_optimizer.py` as a new, pure, deterministic post-processing stage inserted between the existing action-summary/noise-filter response-time views and display. It assigns three independent axes to every candidate SELL/REDUCE trade ‚Äî **Reason** (durable belief-side: Mandatory Risk Reduction / Policy Enforcement / Portfolio Improvement, derived from the already-chosen `action` plus `PolicyEnvelope.violations`, no new AI output), **Necessity** (a pure function of Reason: NECESSARY trades ship in full unconditionally; DISCRETIONARY trades are deferrable), and **Execution Role + Execution State** (STANDALONE/FUNDING_SOURCE/NOT_NEEDED_TODAY √ó FULL/SCALED/DEFERRED ‚Äî what job a trade does in *today's* plan and how much of its own recommended size actually executes). The funding-gap algorithm sorts discretionary candidates ascending by their own full recommended size and walks them greedily, scaling at most one "boundary" candidate down to exactly close the gap ‚Äî least-distortion-from-its-own-plan, not "SELL before REDUCE" or "largest first." Deterministic tie-break: full-release ascending ‚Üí REDUCE preferred over SELL when scaling ‚Üí symbol ascending.
**Reasoning:** Necessity must never be conflated with Execution Role (a trade can be necessary without serving a funding job, and only discretionary trades can ever be scaled ‚Äî a Mandatory Risk exit is never partially executed just because today's cash need is small). Ordering candidates by raw category (SELL/REDUCE) or size-descending, as an early draft of this design did, would partially execute a large intended-full-exit position to cover a small gap ‚Äî a bigger, less explainable distortion from the Belief's own plan than fully executing a small, already-well-fitting candidate instead. The gap between a trade's `full_recommended_amount` and its `executed_amount` is deliberately not routed through `UserExecutionDecision.PARTIAL_EXECUTION` (that field is reserved for a human's later action on top of the already-optimized plan) ‚Äî it feeds Execution Quality metrics (¬ß12: Trade Necessity, Funding Efficiency) and is invisible to Belief Quality grading, which continues to compare against the `ACTIVE_MODEL` shadow's full-compliance simulation, untouched.
**Impact:** `main.py`'s `/analyze/optimizer` and `/optimizer/history/{id}` responses gain an additive `execution_optimization` key (same response-time-view pattern as `action_summary`, backward compatible ‚Äî absent on old unbackfilled rows). `services/funding_source_analysis.py::build_funding_sources()` now delegates its candidate selection to the shared `resolve_funding_gap()` instead of unconditionally including every flagged holding, and gained a `deferred_sources` field. `services/execution_plan.py::build_execution_plan()` now computes buy deployment before funding (the gap must be known first) and derives `funding_actions`/`deferred_funding_actions` from one shared resolution instead of two redundant computations. `RecommendationSnapshot` and `target_allocations` are never touched ‚Äî see `test_execution_optimizer.py` for the founding-example regression test plus ordering/tie-break/edge-case coverage.

### Follow-up fix: position-level policy breach was not detected (2026-07-06)

**Problem:** A design review surfaced that `classify_reason()` only pattern-matched `PolicyEnvelope.violations` strings against the trade's **sector** (`SECTOR_BREACH: {sector}...`), never against its **symbol**. `policy_engine._detect_violations()` also emits a position-level shape, `CONCENTRATION_BREACH: {symbol} at {w}% exceeds {limit}% single-position policy limit`, which never matched. A REDUCE enforcing a live single-position breach was silently classified as discretionary `PORTFOLIO_IMPROVEMENT` instead of `POLICY_ENFORCEMENT` ‚Äî meaning it could be deferred to "Not Executing Today" whenever no funding gap happened to exist, even though the position limit itself was actively being violated.
**Decision:** `classify_reason()` now takes the trade's `symbol` as an explicit parameter and matches a `BREACH` violation line against either the symbol or the sector. Minimal, additive change ‚Äî no new inputs, no architectural change; still pure string-matching against an already-computed violations list (belief-free per ¬ß7).
**Reasoning:** Necessity is a pure function of Reason (¬ß9); a Policy Enforcement trade must ship regardless of cash position. Missing one of the two breach shapes the policy engine actually emits meant a real Priority-2 constraint violation could be masked as a deferrable Priority-5 discretionary trade ‚Äî the opposite of what ¬ß2's objective hierarchy requires.
**Impact:** `execution_optimizer.py::classify_reason(symbol, action, sector, violations)` ‚Äî signature change, one call site (`resolve_funding_gap`) updated. Two new regression tests: a live `CONCENTRATION_BREACH` on the trade's own symbol now classifies as `POLICY_ENFORCEMENT`/`NECESSARY` and ships in full even with `funding_gap == 0`; an unrelated symbol's breach still does not.

## Strategist (L1) Explainability ‚Äî "Stability Review" summary

**Date:** 2026-07-06
**Problem:** A quiet L1 (few or zero swaps ‚Äî intended behavior per its "quality over quantity" conviction bar, see the L1/L2 architecture review the same day) reads to users as "the AI did very little," not "the AI actively reviewed the portfolio and found nothing worth disturbing." L1 had no field explaining its own reasoning ‚Äî `Layer1Result.summary` existed in the schema but was never populated (only L2 wrote to its own `summary`, from `portfolio_assessment`).
**Decision:** Prompt-only change to `_layer1_prompt()` ‚Äî added a required `summary` field: exactly 2 sentences, written in natural investment-strategist voice (what was evaluated, then what cleared the conviction bar and why the rest didn't), with an explicit instruction to vary phrasing run-to-run rather than reuse a fixed template. Also added the same output-budget priority-order fallback L2 already had (`_layer2_prompt`), so a token-constrained response drops `sector_flags`/`top_buys` before it drops `summary` or the actual `swaps`/`priority` decision. Token budget bumped 400‚Üí550 to fit the new field. Frontend: `Layer1Section`'s subtitle changed from `"Recommendations ‚Äî priority: X"` to a static `"Stability Review"`, added a `"Recommendation"`/`"Recommendations"` label above the swap/allocation table, renamed the inline `"Top buys:"` label to a `"Top Watchlist"` section header, and the empty-state string changed from `"No swap proposals from Strategist."` to `"No changes met the conviction bar this cycle."`.
**Reasoning:** None of L1's decision surface changed ‚Äî same conviction thresholds, same max-3-swaps/max-5-top_buys caps, same `no_action` gate, same consensus math (`_consensus_engine`'s `buy_overlap`/`strategist_alignment_score` are untouched). This is purely narrating a decision L1 was already making, in a render slot (`layer.summary` ‚Üí orange callout) that already existed and was already wired through `RecommendationSnapshot` persistence ‚Äî just never populated for L1. Mirrors ¬ß8's "confident silence" principle (a system that explains *why* it did nothing builds more trust than one that says nothing) applied one layer up, to the Strategist itself rather than the Execution Plan.
**Impact:** `agents/optimizer.py::_layer1_prompt()` prompt text only ‚Äî no signature change, no new parameters. `frontend/app/optimizer/page.tsx`'s `Layer1Section`/`SwapTable` ‚Äî copy and layout only, same props, same data. No changes to `_consensus_engine`, `execution_optimizer.py`, or any threshold/scoring logic.

## AI Evaluation M1 ‚Äî Horizon Grading Engine

**Date:** 2026-07-06
**Problem:** `docs/AI_EVALUATION_IMPLEMENTATION_PLAN.md` ¬ß5 M1 needed three cooperating pieces built on M0's schema: a per-recommendation counterfactual (P2), the grading arithmetic that reads it (P3), and EXPIRED semantics for undecided snapshots (P4) ‚Äî none of which exist yet.
**Decision:**
- **P2 (recommendation-level counterfactual):** `services/decision_memory/shadow_tracker.py::create_recommendation_shadow()` ‚Äî a third use of `shadow_type="STATIC_FROZEN"`, keyed to the recommendation itself (`execution_decision_id=NULL`) rather than to a human decision, so every optimizer run gets a return series from day 0 regardless of what the user does. Wired into the existing post-optimizer background thread (`main.py`, beside the ACTIVE_MODEL shadow at line ~2501). Idempotent (one shadow per `recommendation_snapshot_id`); when called against a historical snapshot (backfill path) it also replays the shadow's full daily valuation from stored `PortfolioSnapshot` prices, reusing `repair_shadow_portfolios()`'s existing `_snapshot_price_history`/`_rebuild_shadow_snapshots` routine rather than re-deriving that math.
- **P3 (horizon grading):** `services/evaluation/horizon_grader.py::grade_due_recommendations()` ‚Äî for every snapshot whose age has reached a configured horizon (7/30/90/180 days by default, `evaluation_settings`) and lacks that grade row, reads the recommendation shadow's already-persisted `ShadowPortfolioSnapshot` nearest the horizon date for `return_pct`/`benchmark_return_pct`/`alpha` (zero re-derivation), computes `max_drawdown_pct` via the existing `attribution_engine.compute_max_drawdown()`, and scores directional correctness of each BUY/ACCUMULATE/SELL/REDUCE call against the shadow's own persisted horizon-date prices. Deactivates the shadow once the largest configured horizon is graded, bounding the daily valuation job. Missing shadow / stale valuation (gap > 5 calendar days from the target date) / write failure all skip-with-reason and retry next run ‚Äî never raise, never guess.
- **P4 (EXPIRED semantics):** `services/evaluation/expired_writer.py::write_expired_decisions()` ‚Äî an undecided snapshot becomes an `is_system_generated=True` `UserExecutionDecision(decision="EXPIRED")` row when superseded (a newer sibling snapshot for the same portfolio has any decision, walked newest-to-oldest so same-run cascades work) or aged out (‚â• `evaluation_settings.expiry_days`, default 14), superseded taking priority when both are true.
- Both scheduler jobs are appended to the 17:45 ICT daily chain (`snapshot_scheduler.py`), after `value_all_active_shadows`, EXPIRED writer first so a snapshot that ages out today is gradable as such starting today.
- **Backfill:** `python manage.py backfill_recommendation_grades --all|--portfolio ID` ‚Äî one-shot CLI (mirrors `verify_snapshots` conventions) that creates missing recommendation shadows (with historical replay) and grades every now-mature horizon, so the Evaluation phase does not launch empty for existing users. Idempotent by construction (shadow creation and grading are both create-if-missing).
**Reasoning:** Reusing the shadow's own persisted daily valuation (rather than a fresh price lookup at grading time) keeps grading and shadow valuation from ever disagreeing about what a recommendation was "worth" on a given day ‚Äî a second source of truth for the same number was the exact anti-pattern PLAN ¬ß4.6 rules out. Cascading EXPIRED newest-to-oldest in one pass avoids a multi-day lag for chains of ignored recommendations.
**Impact:** `services/decision_memory/shadow_tracker.py` (+`create_recommendation_shadow`), `services/evaluation/{horizon_grader,expired_writer}.py` (new), `snapshot_scheduler.py`, `manage.py` (+`backfill_recommendation_grades`), `models/database.py` (`is_system_generated`, used here for the first time). 17 new tests in `tests/test_horizon_grader.py`, all passing. M2 (plan grading + execution analysis) is next.

## AI Evaluation M2 ‚Äî Plan Grading & Execution Analysis

**Date:** 2026-07-06
**Problem:** `docs/AI_EVALUATION_IMPLEMENTATION_PLAN.md` ¬ß5 M2 needed a day-0 plan-quality composite (G4) and a plan-vs-actual execution comparison (half of G5) ‚Äî both scored deterministically off data the pipeline already produces, without touching the optimizer or execution_optimizer.py.
**Decision:**
- **Plan grade:** `services/evaluation/plan_grader.py::compute_plan_grade()` ‚Äî a 0-100 composite over four documented, weighted sub-scores: **necessity** (30%, share of SELL/REDUCE candidates that are Required-tier or actively funding today), **funding efficiency** (30%, THE BH-INCIDENT METRIC ‚Äî of the total amount discretionary candidates proposed to release, what fraction was actually needed to close the funding gap; a REDUCE proposing far more than the gap requires scores near the floor even though execution_optimizer.py already defers the excess), **turnover proportionality** (20%, reads the already-computed `TURNOVER_BREACH` violation_detail, no new turnover math), **explanation completeness** (20%, every candidate trade carries a non-empty reason/necessity/execution_role/execution_state/note, or the no-trade case carries a portfolio_assessment/no_action_summary). Re-derives the plan via the existing `optimize_execution()` on stored `target_allocations`/`cash_balance`/`active_policy.violations` ‚Äî same inputs, same grade, always (asserted in tests). `grade_pending_plans()` persists `grade_kind="PLAN"` rows from the post-optimizer background thread (no maturity wait needed) and via the M1 backfill CLI (extended, not duplicated).
- **Execution analysis:** `services/evaluation/execution_analyzer.py::compute_execution_analysis()` ‚Äî a pure function comparing the re-derived plan against `Transaction` rows linked via `execution_decision_id` (P5): **timing** delta (recommendation-date price vs. share-weighted fill price), **size** delta (planned vs. executed amount), **funding fidelity** (planned FUNDING_SOURCE trades vs. actual linked sells), **completeness** (fraction of the plan with a linked transaction). A decision with zero linked transactions returns `status="unavailable"`, `score=None` ‚Äî never a fabricated number; partial matches renormalize the composite over only the measurable components and mark `status="partial"`. No DB writes ‚Äî a read-only comparison, left for M3 to expose via `GET ‚Ä¶/execution/{decision_id}`.

Metadata-only, defaults to `None`, omitted from every existing call site untouched.
**Reasoning:** Funding efficiency deliberately grades the RECOMMENDATION's own proportionality, not just whether execution_optimizer.py caught the excess ‚Äî a plan that requires heavy correction is a worse plan even when the correction succeeds, which is what separates "plan quality" from "execution-optimization quality" as independent, addressable failure points (PLAN ¬ß12, three-lens independence). Execution analysis returning `None` rather than a fabricated composite when nothing is linked follows PLAN ¬ß4.7/¬ß4.8 literally ‚Äî an evaluation number must never imply confidence the data doesn't support.
**Impact:** `services/evaluation/{plan_grader,execution_analyzer}.py` (new), `services/portfolio_transactions.py`, `main.py` (buy/sell bodies + endpoints, `_tx_row`, post-optimizer background PLAN-grading thread), `manage.py` (`backfill_recommendation_grades` now also grades PLAN). M3 (aggregation APIs + verdict composer) is next.
**Impact:** `services/decision_memory/shadow_tracker.py` (+`create_recommendation_shadow`), new `services/evaluation/horizon_grader.py` and `services/evaluation/expired_writer.py`, `main.py` (background-thread wiring only, no new endpoints), `services/snapshot_scheduler.py` (+2 calls in the existing daily chain), `manage.py` (+`backfill_recommendation_grades` subcommand). 17 new unit tests (`tests/test_horizon_grader.py`, in-memory SQLite against the real ORM models) covering maturity boundaries, append-only idempotency, skip-with-reason paths, deactivation, directional scoring, EXPIRED supersession/aging, and shadow-creation idempotency ‚Äî all passing. ## Gap A Correctness Patch ‚Äî Price-Source Unification + Event-Sequence Alignment

**Date:** 2026-07-06
**Problem:** An architecture review of M6 (requested before M7 began) asked why `ideal_return_pct` swings far more sharply than `ACTIVE_MODEL`'s own return on live dev data (portfolio 2: ~-85% vs ~-20%, Gap A ~-65%), and whether Gap A was actually isolating execution friction or an artifact of the two hypothetical portfolios being computed two different ways. Re-reading the code (not just re-running the tests) surfaced two real, distinct correctness defects, neither of which had been caught by M6's unit tests or its live-verification pass.
**Root cause 1 ‚Äî price source was never unified.** `compute_ideal_series` (Ideal) prices holdings from `_snapshot_price_history` ‚Äî the daily-close prices recorded once/day in `PortfolioSnapshot.holdings_json`. But `compute_portfolio_attribution`'s `ai_model_shadow.return_pct` (what M6 fed into Gap A for the AI side) is read straight from persisted `ShadowPortfolioSnapshot.total_value`/`daily_return_pct` rows, which `shadow_tracker.value_shadow_portfolio` populates from `_fetch_cached_prices` ‚Äî a live AgentCache "technical" price read (15-min TTL) captured at whatever moment the daily valuation job happened to run, falling back to the same snapshot archive only on a cache miss. Both ultimately trace to yfinance, but at different times of day through different caches ‚Äî day-over-day divergence between the two compounds through `_compute_twr`'s daily chaining, contaminating Gap A with price-source noise rather than friction.
**Root cause 2 ‚Äî the two sides don't cover the same time horizon.** Ideal replays every `RecommendationSnapshot` from `period_start` forward (21 snapshots over 90 days on live portfolio 2, confirmed via direct DB query). The `ACTIVE_MODEL` shadow replays the identical trigger (`main.py`'s optimizer-run handler writes the snapshot and rebalances the shadow in the same request ‚Äî confirmed by reading both call sites; the daily scheduler only values shadows, never rebalances them), so the two sequences are aligned *in principle*. But on live portfolio 2, the shadow's `inception_date` had been reset to 2026-06-23 (Phase 4C.6's `reset_active_model_inception`), meaning its own track record only covers the last ~13 days of the 90-day window, while Ideal was replaying the full 90 days including a volatile stretch (2026-05-25 ‚Üí 2026-06-22) the shadow never experienced. Comparing a 90-day Ideal return against a 13-day AI return silently conflated "different time horizons" with "friction cost" ‚Äî the actual dominant driver of the -65% Gap A, confirmed by re-running against live data after fixing root cause 1 alone (Gap A barely moved, to -64.9%) versus after fixing root cause 2 (Gap A moved to +8.8%).
**Decision:**
- **`ideal_series.py::_revalue_ai_portfolio_with_canonical_prices()` ‚Äî new.** Reads the active `ACTIVE_MODEL` shadow's own persisted `ShadowPortfolioSnapshot.holdings_json` (what it held, and when it rebalanced ‚Äî untouched, read as-is) but revalues those holdings through the identical canonical price archive Ideal uses, instead of trusting the row's own `total_value`. `compute_three_portfolios` now sources `ai_return`/`ai_portfolio.*`/the chart's `ai` line from this function, never from `compute_portfolio_attribution`'s `ai_model_shadow` for Gap A purposes. `compute_portfolio_attribution`'s own `ai_model_shadow` figure is untouched and remains correct for its existing consumers (Scorecard, S4, S5, the attribution waterfall) ‚Äî a legitimately different question ("what has the persisted paper account actually returned") from Gap A's ("controlling for identical prices, how much is friction").
- **`ideal_series.py::_return_over_window()` / `_index_at_or_before()` ‚Äî new.** `compute_three_portfolios` now computes `comparison_start = max(ideal_first_date, ai_first_date)` and recomputes both sides' `return_pct`/`max_drawdown_pct`/`annualized_volatility` over `[comparison_start, period_end]` before deriving `gap_a` ‚Äî so the `ideal`/`ai_portfolio` blocks shown alongside Gap A are always arithmetically consistent with Gap A itself, and a shadow-inception-reset mid-window can no longer silently produce a horizon-mismatched gap. Gap B (`regret_score`, Actual vs AI) is untouched ‚Äî Actual is a real account, so comparing it against the shadow's own official live-priced valuation remains the correct real-world comparison.
- **`ideal_series.py`'s own `compute_ideal_series()` public contract is unchanged** ‚Äî full `period_days`, same status vocabulary, same friction-free definition (immediate execution, zero cost, perfect fills, unlimited liquidity, no human intervention). Only `compute_three_portfolios`'s internal Gap A comparison was touched, per the explicit instruction not to redesign Ideal.
**Reasoning:** Both defects were about *comparability*, not about Ideal's own definition ‚Äî Ideal's friction-free methodology (documented once, Risk R6) is unchanged; what changed is ensuring the AI Portfolio side of the comparison is priced identically and windowed identically before the subtraction happens. Neither fix required touching `compute_portfolio_attribution`, `shadow_tracker.py`, or any other already-shipped M0‚ÄìM6 consumer, keeping blast radius to the one function (`compute_three_portfolios`) that owns Gap A.
**Impact:** `backend/services/evaluation/ideal_series.py` only (+`_revalue_ai_portfolio_with_canonical_prices`, +`_return_over_window`, +`_index_at_or_before`; `compute_three_portfolios` rewired internally, same public response shape ‚Äî no frontend or API contract change, `tsc --noEmit` stays clean with zero edits). 6 new backend tests in `tests/test_ideal_series.py` (now 18 total in that file), all passing: canonical-price revaluation in both directions (flat-price and appreciating-price scenarios), a legacy-row-without-`holdings_json` status test, a no-snapshots-in-window status test, and the event-sequence-alignment regression (a simulated inception reset with an early price crash Ideal alone would have replayed, collapsing a would-be -90 Gap A artifact to ~0 once aligned). 92 tests pass across every directly related test file (`test_ideal_series`, `test_attribution_waterfall`, `test_verdict_composer`, `test_scorecard`, `test_human_vs_ai_scoreboard`, `test_regime_attribution`, `test_horizon_grader`, `test_execution_ledger`, `test_plan_grader`). Full-suite run reproduces the same pre-existing, order-fragile failure set documented since M0/Phase 4C.1 (confirmed by re-running the specific failing files in isolation ‚Äî e.g. `test_optimizer_pipeline.py`'s 4 failures are a pre-existing `_consensus_engine()` signature mismatch, unrelated to this patch and unaffected by it). Verified against the live dev backend, portfolio 2: before this patch, `three_portfolios` reported `ideal: -84.5%`, `ai_portfolio: -19.6%`, `gap_a: -64.9%`; after, `ideal: -10.85%`, `ai_portfolio: -19.65%`, `gap_a: +8.79%` ‚Äî both sides now measured from the shadow's actual 2026-06-23 inception forward (visible in the chart's `ai` line, `null` before that date, matching the DB's own `shadow_portfolios.inception_date`), and Gap A flipped from a wildly exaggerated negative number to a small, plausible positive one consistent with its own composed interpretation ("the price of practical execution").

## Paper Portfolio Cash-Leak Fix (Accounting Correctness Milestone 1)

**Date:** 2026-07-07
**Problem:** A follow-up architecture review (requested against a specific live example: portfolio 4's 90-day chart showing Ideal at -24.5% and AI Portfolio at -51.7%, both far worse than the July 6 Gap A patch alone could explain) traced both declines to a defect distinct from anything that patch touched. Every `RecommendationSnapshot.projected_allocations_json` for the affected portfolios intentionally sums `target_weight` to less than 100% ‚Äî confirmed as the optimizer's own cash floor, not a data defect (`OPTIMIZER_PHILOSOPHY.md` ¬ß2 Priority 7 "Idle-Cash Efficiency"; `policy_engine.py`'s `min_cash_pct` enforcement, which actively raises a `POLICY_VIOLATION` if implied cash falls *below* the floor). But `_resolve_shares_from_weights()` (`shadow_tracker.py`, shared by `compute_ideal_series` and `create_active_model_shadow`) computed `market_value = target_weight / 100 * total_portfolio_value` per holding and returned only the holdings list ‚Äî the unallocated remainder (2‚Äì18% across the affected portfolio's history, frequently ~5%) was never written to any field, including the pre-existing but always-0.0 `ShadowPortfolio.paper_cash_balance` column. Every full rebalance therefore silently deleted that fraction of NAV, compounding multiplicatively: arithmetic reconstruction of the live example's 6 aligned-window rebalances (weight-sums 95/95/95/97.1/93.3/93.7%) predicted a -27.2% decline from leakage alone versus Ideal's actual -24.5% aligned return (real prices were mildly net-positive) ‚Äî the leak, not friction or genuine market performance, was the dominant driver of both lines.
**Decision:** Scoped to Milestone 1 of the approved accounting design review only (shared replay engine correctness; historical regeneration, Gap A/Opportunity Cost/Scorecard wiring, and any API/frontend change are explicitly deferred to later milestones):
- **`_resolve_shares_from_weights()` now returns `(holdings, cash)`**, where `cash = max(0.0, (100 - Œ£target_weight) / 100 * total_portfolio_value)` ‚Äî the unallocated residual as explicit paper cash, never dropped. Every one of its 5 call sites (`create_static_frozen_shadow`, `create_active_model_shadow` √ó2 ‚Äî inception and rebalance, `create_recommendation_shadow`, `compute_ideal_series`) now persists/carries `cash` forward instead of hardcoding `paper_cash_balance=0.0` or discarding the return value.
- **`create_active_model_shadow`'s `running_nav` computation** now adds `existing.paper_cash_balance` back in on every path (previously only the `old_holdings` branch did; the empty-holdings `else` branch did not), so cash is never dropped even when a shadow temporarily holds no positions.
- **`compute_ideal_series`'s replay loop** carries a `cash` accumulator across iterations: each rebalance's NAV basis is now `equity(old_holdings) + cash` (previously `equity(old_holdings)` alone ‚Äî the exact leak), and every valuation adds `cash` back in (cash earns zero return between rebalances, per the approved design).
- **New shared invariant, `assert_nav_conserved()` / `NavInvariantError`** (`shadow_tracker.py`): asserts `equity + cash == expected_nav` within a `0.01`-or-relative tolerance (mirroring `PORTFOLIO_CALCULATION_RULES.md` ¬ß9's existing NAV-reconciliation tripwire convention) immediately after every `_resolve_shares_from_weights` call across both engines. Raises loudly (never silently continues) per `ENGINEERING_PRINCIPLES.md`'s Failure Handling section, and is explicitly reusable by any future paper-portfolio engine.
- **Target weights are never normalized to 100%, the optimizer/policy engine/execution planning are untouched**, and `RecommendationSnapshot` rows are untouched ‚Äî the cash floor itself is correct and intentional (per the review); only its downstream replay was defective.
**Reasoning:** This is a correctness bug in a shared accounting function, not a design change ‚Äî the unallocated weight was always real cash by the optimizer's own policy, and Design Principle 1 (`PORTFOLIO_CALCULATION_RULES.md` ¬ß9's NAV invariant, `total_value = cash_balance + equity_value`, always, exactly) was never actually being enforced for the two paper-portfolio engines the way it already is for the real ledger-backed ones. Fixing the one shared function (rather than patching each caller ad hoc) satisfies `ENGINEERING_PRINCIPLES.md`'s Single Source of Truth and Reuse-Before-Create principles, and the new invariant assertion converts a previously invisible defect into one that cannot recur silently.
**Impact:** `backend/services/decision_memory/shadow_tracker.py` (+`NavInvariantError`, +`assert_nav_conserved`, `_resolve_shares_from_weights` signature change to `tuple[list[dict], float]`, all 3 internal call sites updated) and `backend/services/evaluation/ideal_series.py` (`compute_ideal_series` cash carry-forward, both call sites updated) only. ## Historical Regeneration + Replay Validation Engine (Accounting Correctness Milestone 2)

**Date:** 2026-07-07
**Problem:** Milestone 1 fixed the shared replay functions so *future* rebalances stop leaking cash, but left every already-persisted `ShadowPortfolio.paper_cash_balance` value and `ShadowPortfolioSnapshot` row exactly as they were before the fix ‚Äî still reflecting the leak. Milestone 2 authorizes regenerating that derived history through the corrected engine, plus adding reusable replay-determinism validation so future simulation engines inherit the same guarantee.
**Decision:** Scoped to Milestone 2 of the approved accounting design review only (historical regeneration + replay validation; no optimizer/policy/execution/API/frontend change; no methodology redesign):
- **`verify_deterministic_replay(fn, *args, **kwargs)`** (`shadow_tracker.py`) ‚Äî reusable helper that calls a pure replay function twice and deep-compares the results within tolerance, raising `ReplayNonDeterministicError` (loud, never silent) on divergence. Reused to prove `_replay_active_model_series` and `compute_ideal_series` are both deterministic.
- **`regenerate_static_shadow(db, shadow_id)`** ‚Äî re-derives `paper_cash_balance` for one STATIC_FROZEN or recommendation-keyed shadow directly from its own already-stored per-holding `target_weight_pct` values (never re-derived from a live recommendation/decision/portfolio state, which would not reproduce history), then replays its snapshot history via the existing `_rebuild_shadow_snapshots`.
- **`_replay_active_model_series` / `regenerate_active_model_shadow(db, portfolio_id)`** ‚Äî ACTIVE_MODEL holdings change at every rebalance, so its history is replayed rebalance-by-rebalance from the shadow's immutable `inception_date`/`inception_value` forward, walking every `RecommendationSnapshot` exactly as `create_active_model_shadow` does live, with `assert_nav_conserved` checked on **every single regenerated day** (not only rebalance points) ‚Äî the explicit Milestone 2 requirement. `inception_date`/`inception_value` are never touched.

- **CLI: `manage.py regenerate_paper_portfolios`** (`--portfolio`/`--all`, `--commit`, `--backup`, `--yes`) ‚Äî defaults to dry run; records before/after `ShadowPortfolio`/`ShadowPortfolioSnapshot`/`AttributionMetric` counts; refreshes each regenerated portfolio's `AttributionMetric` row via the existing idempotent `compute_portfolio_attribution` upsert; `--backup` exports affected rows to JSON before committing (mirrors `portfolio_rebuilder.py`'s pre-commit backup convention).
- **A separate, pre-existing correctness bug was discovered and deliberately NOT fixed here**: 12 `STATIC_FROZEN` shadows (portfolio 2: shadow ids 8, 12, 14, 16, 19, 21, 23, 26, 37, 38, 41; portfolio 4: shadow id 32) fail the NAV invariant because `create_static_frozen_shadow`'s "actual portfolio holdings" fallback branch computes each holding's weight as `current_market_value / total_portfolio_value` using the *stale* `total_portfolio_value` recorded on the linked `RecommendationSnapshot` as the denominator, instead of self-normalizing against the sum of current market values ‚Äî so whenever real market value has drifted from that stale figure, stored weights sum to >100% (up to ~110% observed). Repair is deliberately deferred to a future, separately-scoped change ‚Äî the principled fix corrects `inception_value` to match the actual frozen holdings, not the reverse, and must not be bundled into this milestone.
**Reasoning:** Regenerating derived rows through the now-corrected engine is safe and additive as long as the invariant that guards correctness is checked on every row, not just spot-checked ‚Äî the 12-shadow anomaly is direct proof the guard works as designed rather than rubber-stamping bad data. Deferring the fallback-path bug keeps this milestone's blast radius to exactly what Milestone 1 already fixed (the shared replay engine), per `ENGINEERING_PRINCIPLES.md`'s "no duplicated business rules" and the explicit instruction not to bundle unrelated correctness issues into one change.
**Impact:** `backend/services/decision_memory/shadow_tracker.py` (+`ReplayNonDeterministicError`, +`verify_deterministic_replay`, +`regenerate_static_shadow`, +`_replay_active_model_series`, +`regenerate_active_model_shadow`, +`regenerate_portfolio_paper_history`), `backend/manage.py` (+`regenerate_paper_portfolios` CLI subcommand and its helpers), `backend/tests/test_shadow_regeneration.py` (new, 13 tests: determinism pass/fail/tolerance, static-shadow cash restoration/dry-run/idempotency, multi-day multi-rebalance ACTIVE_MODEL replay/determinism/idempotency/dry-run, a documented edge case where a literal full-liquidation snapshot is safely skipped rather than misapplied by the history-scanning replay, portfolio-level orchestration, and `compute_ideal_series` replay determinism). ## Three Portfolios Integration Consistency Fixes (Accounting Correctness C5)

**Date:** 2026-07-07
**Problem:** An architecture review of the live Three Portfolios screen (post-Milestone-2) found two integration-level inconsistencies, neither a flaw in the (already-approved) accounting model or replay engine themselves:
- **Issue A ‚Äî `regenerate_active_model_shadow` left `holdings_json` internally inconsistent.** It correctly recomputed and wrote `total_value`/`return_pct_since_inception`/`daily_return_pct` on an already-existing `ShadowPortfolioSnapshot` row, but never touched that row's `holdings_json` ‚Äî which on live data was whatever the pre-Milestone-1 daily-valuation scheduler had written (a leak-era share composition). Downstream, `ideal_series.py::_revalue_ai_portfolio_with_canonical_prices` (the approved Gap A price-source-unification path, "Gap A Correctness Patch") reads `holdings_json` ‚Äî not `total_value` ‚Äî to revalue the shadow through the canonical price archive. On live portfolio 4 this produced `ai_portfolio.return_pct = -48.33%` and `gap_a = +51.85%`, even though the shadow's own correctly-regenerated `total_value` trajectory (and `compute_portfolio_attribution`'s `ai_model_shadow.return_pct`, which reads `total_value`/`daily_return_pct` directly) already showed a sane `+3.52%`.
- **Issue B ‚Äî the Three Portfolios chart's "You" line used a raw NAV ratio.** `compute_three_portfolios` built the chart's `actual` series from `PortfolioSnapshot.total_value` reindexed to 100, while the adjacent summary card's `actual.return_pct` already used the cash-flow-adjusted TWR chain (`attribution_engine._actual_portfolio_metrics`, chaining `investment_return_pct`, established at Phase 3B.8). A mid-window deposit inflates raw `total_value` without being investment return; on live portfolio 4 this showed the chart ending near +22% against a +4.6% summary card for the same window.
**Decision:** Scoped to exactly these two integration defects ‚Äî no optimizer/policy/algorithm change, no redesign of `_revalue_ai_portfolio_with_canonical_prices`'s canonical-price methodology or of the TWR formula itself (both remain exactly as previously approved):
- **`_replay_active_model_series`** (`shadow_tracker.py`) now captures each day's already-computed enriched holdings (the `_compute_paper_value` return value it previously discarded via `_`) into its `series` rows, in the identical shape `value_shadow_portfolio` already persists to `holdings_json` for the live daily-valuation path.
- **`regenerate_active_model_shadow`** now writes that same per-date `holdings_json` on both the `existing`-row and new-row upsert branches ‚Äî so `holdings_json`, `total_value`, `daily_return_pct`, and `return_pct_since_inception` always describe one consistent regenerated state, never a stale pre-regeneration composition paired with a corrected total.
- **`services/analytics/attribution_engine.py::compute_actual_indexed_series()` ‚Äî new.** Chains `PortfolioSnapshot.investment_return_pct` with the identical multiplicative step `_compute_twr` already uses for the summary card's return figure, exposed as a running per-date index instead of one final number, so a chart and its card can never disagree. Falls back to a raw two-point NAV ratio only when the window has zero adjusted returns at all (legacy pre-Phase-3B.8 rows) ‚Äî the same fallback `_actual_portfolio_metrics` uses.
- **`compute_three_portfolios`** now sources `actual_dated` from `compute_actual_indexed_series` instead of a raw `PortfolioSnapshot.total_value` query.
- **`_reindex_to_100` gained a `rebase: bool = True` parameter.** The existing rebase-to-first-observed-axis-date behavior (correct for `ideal`/`ai`/`benchmark`, each already self-normalized to exactly 100 at its own start, where a second rebase is a no-op) is unchanged and is the default. `compute_three_portfolios` now calls it with `rebase=False` for `actual`, since `compute_actual_indexed_series`'s TWR-chained output is already absolute-scaled and may legitimately not equal 100 at the first date the shared multi-series axis observes it ‚Äî rebasing it a second time would renormalize that value back to 100 and silently erase the return it represents.
**Reasoning:** Both defects were integration gaps ‚Äî a value computed once and either not propagated to every field that must agree with it (Issue A) or propagated through a different, non-equivalent formula than its sibling display already used (Issue B) ‚Äî not flaws in the underlying accounting model, the replay engine, or the TWR methodology, none of which were touched. Reusing `_compute_twr`'s exact formula (Issue B) rather than inventing a second return calculation satisfies `PORTFOLIO_CALCULATION_RULES.md` ¬ß10 ("never recompute return inside an analytics/attribution module") and `ENGINEERING_PRINCIPLES.md`'s Single Source of Truth; capturing an already-computed value instead of adding new computation (Issue A) is the minimal fix for an internal-consistency gap, not a new business rule.

## Scorecard "Win Rate" Was a Mislabeled, Duplicated Metric

**Date:** 2026-07-07
**Problem:** S1 Scorecard's Outcome Quality card (`Did it work?`) showed a "Win rate" stat (e.g. "+81% ¬∑ 35 AI ¬∑ 8 you") directly beneath cumulative portfolio returns (You/AI Portfolio/Ideal/Benchmark). Users naturally read "Win rate" in that position as "who is winning overall" ‚Äî i.e. a portfolio-level performance comparison. It is not: `outcome.win_rate` is `compare_human_vs_ai()`'s per-`UserExecutionDecision` count (how many individual decisions, each valued from its own `executed_at` date via a live ad-hoc shadow lookup, the AI's shadow beat the actual portfolio on). Mixing a decision-count statistic into a lens whose other four numbers are cumulative-return percentages over one shared window was the root confusion, not just the label.
**Decision:**
- Removed the "Win rate" stat from the Outcome Quality card entirely rather than relabeling it in place. It was not moved because a better-sourced equivalent already exists: S5 Human vs AI's "Scoreboard" (`you_beat_ai`/`ai_beat_you`/`ties`) is grade-row-sourced (`compute_scoreboard`, M5) rather than the live ad-hoc valuation `compare_human_vs_ai()` uses ‚Äî duplicating the concept here with a second, weaker-sourced number would have created two answers to "who wins more often" (a Single Source of Truth violation this phase has otherwise been careful to avoid ‚Äî see M6's Gap A/B reuse).
- Replaced it with **Net Opportunity Cost** (`outcome.net_opportunity_cost`, already computed by M5's `opportunity_cost.py` and already typed on `OutcomeLens`, but never rendered on this card): a portfolio-level percentage ‚Äî the summed counterfactual effect of every divergence (REJECTED/PARTIAL/OVERRIDE/ignored-EXPIRED) vs. what full compliance would have returned ‚Äî so it stays consistent with the other four Outcome Quality stats (a %, not a count) and answers the same "did it work" question the lens is titled for. Rendered via the existing `GapAnnotation` component (same honesty conventions as Gap A: unavailable-with-reason, maturing-with-pending-count), plus a link to the full Opportunity Cost ledger (`/ai-analytics/opportunity-cost`) for the per-decision detail. `outcome.win_rate` itself was left untouched in the backend ‚Äî `compose_scorecard_verdict`'s `gap_b_n` sample-size gate still reads `win_rate.n`, and that wiring is unrelated to this display bug.
- **Why not "Head-to-head record" (rename in place)?** That was the fallback the request offered if the metric were "not sufficiently meaningful" to duplicate elsewhere ‚Äî since S5 already shows the more rigorously-sourced version, renaming and keeping a second, ad-hoc-valuation copy here would still leave two numbers answering the same question with different methodologies.
**Reasoning:** A card's metrics should share one unit and one time basis, or a reader will average across them without realizing they're incompatible. Net Opportunity Cost was chosen over the other candidates raised (Alpha vs Benchmark, Information Ratio, Tracking Difference) because it is the only one already computed, already honesty-gated (graded/maturing/cold_start), and already scoped to exactly "the human's own decisions vs. the AI's" ‚Äî the same question Outcome Quality is asking, at the same portfolio-percentage unit as its siblings.
**Impact:** `frontend/app/ai-analytics/(hub)/page.tsx` only (Outcome Quality card). No backend change ‚Äî `net_opportunity_cost` and `win_rate` were both already computed and typed; this was a frontend rendering fix. `tsc --noEmit` clean.

## Platform Architecture Constitution v1.1 ‚Äî Governance Hierarchy and Canonical Vocabulary (Additive Amendment)

**Date:** 2026-07-10
**Problem:** Constitution v1 (`docs/architecture/platform_architecture.md`, ratified the same day) defined the laws, layers, and nine domains, and ¬ß10 defined its own amendment process and pairwise precedence with the two Domain Constitutions ‚Äî but it did not state a complete precedence hierarchy across all of the platform's written artifacts (constitution, domain constitutions, ADRs, design documents, implementation docs, source code), leaving conflicts between levels to be resolved ad hoc. Separately, the platform had a vocabulary document (`docs/GLOSSARY.md`) referenced by the Architecture Handbook as shared vocabulary, but nothing designated it canonical or governed how terms are added, reserved, or redefined ‚Äî and the constitution had introduced load-bearing terms (the nine domain names, the three gates, witness/authority, observer plane, provenance, proposal, derivation) that the glossary did not contain.
**Decision:**
- Added **¬ß11 Architecture Governance** to the constitution: a six-level precedence hierarchy (Platform Architecture ‚Üí Domain Constitutions ‚Üí ADRs ‚Üí Technical Design Documents ‚Üí Implementation Documentation ‚Üí Source Code) with six governance rules (G1‚ÄìG6): higher states intent / lower states reality; lower may refine but never weaken; silence delegates downward; conflict is a defect resolved upward, never by recency; each level amends by its own mechanism (ADRs superseded, never edited ‚Äî Law 2 applied to the platform's record of itself); code is never precedent.
- Added **¬ß12 Canonical Vocabulary** to the constitution: `docs/GLOSSARY.md` is designated the canonical vocabulary (Law 9 applied to language) with four rules (V1‚ÄìV4): one term, one meaning, one home; new nouns registered before relied upon; constitutional terms are reserved and changing their *meaning* is a ¬ß10 amendment; the glossary is not a governance level but the shared language of all six.
- **Deliberately did not create a second vocabulary document.** The review brief proposed "a separate canonical vocabulary (Glossary) document"; a glossary already existed, and creating another would have violated the very one-home-per-concept rule being codified. The existing `docs/GLOSSARY.md` was extended **append-only** with the constitution's missing terms (Ledger, Canonical Ledger Event, Derivation, Provenance, Proposal, Witness, Gate, Observer Plane, Domain, Platform Domains, Asset Definition, Domain Constitution); no existing entry was modified.
- Both amendments are purely additive: Sections 1‚Äì10 (Constitution v1) are byte-identical to ratification; a version line in the preamble records v1 ‚Üí v1.1.
**Reasoning:** ¬ß10's amendment discipline is only enforceable if everyone agrees which document outranks which ‚Äî the governance hierarchy makes the constitution's authority explicit rather than implied, and G6 ("code is never precedent") closes the most common erosion path, where implementation drift is retroactively accepted as design. The vocabulary rules exist because a platform that lets one word mean different things in different documents will eventually let it mean different things in different engines; reserving the constitutional terms prevents the domain names and gate vocabulary from being diluted by casual reuse.
**Impact:** `docs/architecture/platform_architecture.md` (+¬ß11, +¬ß12, +version line in preamble ‚Äî no existing section modified), `docs/GLOSSARY.md` (+12 entries, append-only), `docs/architecture/README.md` (handbook version v5.0 ‚Üí v5.1 per its own material-revision rule). No code change, no schema change, no behavior change.

## M14 ‚Äî Runtime Enforcement Stage R2, First Controlled Gate

**Date:** 2026-07-13
**Problem:** M13 produced a complete, honest per-`AssetType` enforcement decision table but the runtime still decided nothing ‚Äî Stage R1/R1.5 consultations were purely observational (M11/M12) or purely reporting (M13). This milestone asks for the *first* place a runtime-driven decision is allowed to change an outcome, scoped narrowly: one consumer (`asset_registry.mint()`), reversible, off by default, and driven only by the M13 decision table ‚Äî never by inferring policy from a missing definition.
**Decision:**
- New `backend/services/asset_definitions/enforcement_gate.py`: `evaluate_mint_enforcement(asset_type, *, mode=None, decisions=ENFORCEMENT_DECISIONS)` ‚Äî a pure function, no persisted state. It looks up the `AssetType`'s row in `ENFORCEMENT_DECISIONS` and maps `future_action` to an `intended_action` (`REJECT` ‚Üí `BLOCK`; `NOT_APPLICABLE`/`PRESERVE`/`MIGRATE` ‚Üí `ALLOW` ‚Äî a `MIGRATE` verdict is explicitly *not* authorization to reject, per the brief: "Do not silently reject unless the M13 decision explicitly authorizes it"). Three `EnforcementMode` values ‚Äî `OFF`/`SHADOW`/`ENFORCE` ‚Äî gate whether `intended_action` becomes the `effective_action`: only `ENFORCE` ever lets `BLOCK` through; `OFF` and `SHADOW` always resolve to `ALLOW` regardless of what the decision says, so `blocked=True` is reachable only via `ENFORCE`. Mode resolves per call: an explicit `mode=` argument wins, otherwise the `ASSET_MINT_ENFORCEMENT_MODE` environment variable, otherwise `OFF` ‚Äî an unrecognized env value logs a warning and falls back to `OFF` rather than raising, keeping the default path incapable of accidentally enforcing anything.
- `backend/services/asset_registry.py::mint()` gains one new keyword-only parameter, `enforcement_mode: Optional[EnforcementMode] = None` (backward compatible ‚Äî every existing caller, including `registry_service.mint_asset() ‚Üí core.mint()`, already calls `mint()` with keyword arguments after `claim` and needed no changes). After the existing M12 shadow consultation (untouched), `mint()` now also calls `evaluate_mint_enforcement()`, logs the full outcome at `INFO` (asset_type, mode, gap_type, intended_action, effective_action, reason ‚Äî the brief's "Required Observability" list, as structured fields, not just log text), and raises `AssetRegistryError` if and only if `outcome.blocked` is `True`.
- As of this milestone, no row in the real `ENFORCEMENT_DECISIONS` table (M13) has `future_action=REJECT` ‚Äî so `ENFORCE` mode against real data is a documented no-op: every `AssetType` member still mints successfully under `ENFORCE` today (asserted directly, parametrized over all nine members, both at the `evaluate_mint_enforcement()` level and end-to-end through `mint()`). This is intentional, not a gap: the brief's non-goals explicitly forbid enforcing all missing definitions or rejecting all undefined `AssetType`s in this milestone, and the M13 decision table is the only place a future `REJECT` may be authorized. The blocking path itself is proven with a synthetic decisions tuple (one row's `future_action` swapped to `REJECT` in a test-local copy, never in the real table) rather than by adding a real `REJECT` decision.
- 48 new tests in `test_asset_registry_enforcement.py` covering: existing-behavior preservation (all nine types mint successfully under `OFF` and under the unset-argument default); shadow behavior (a synthetic `REJECT` computes `intended_action=BLOCK` but `effective_action` stays `ALLOW`, `blocked=False`); enabled enforcement (a synthetic `REJECT` on `ETF` blocks only `ETF` ‚Äî every other binding in the same synthetic table, and `mint()` itself for a different `asset_type`, is unaffected); rollback (the identical synthetic decision flips `blocked` `False‚ÜíFalse‚ÜíTrue‚ÜíFalse` purely by changing `mode=` across `OFF‚ÜíSHADOW‚ÜíENFORCE‚ÜíOFF`, no state carried between calls); decision consistency (real `ENFORCEMENT_DECISIONS` has zero `REJECT` rows today, and `ENFORCE` mode against it blocks nothing for any of the nine `AssetType` members); observability (`caplog` asserts the structured log line's fields); and configurability (an explicit `enforcement_mode=` argument to `mint()`/`evaluate_mint_enforcement()` overrides the environment variable, an unrecognized environment value falls back to `OFF`).
**Reasoning:** "Enforcement must follow decisions, not infer policy" was taken as the load-bearing constraint: the gate function's only decision input is `EnforcementDecision.future_action`, read verbatim from M13's hand-authored table ‚Äî it does not independently decide that "missing definition" implies "reject," which is exactly the shortcut the brief's Rollout Safety and Enforcement Rules sections warn against (`MIGRATE` is not `REJECT`). Making `evaluate_mint_enforcement()` a pure function of `(asset_type, mode, decisions)` with no persisted state was chosen specifically to make "Rollback" (a required test category) trivial and structurally guaranteed rather than something a stateful flag/cache could get wrong. Testing the actual `BLOCK` path against a synthetic decisions tuple, rather than adding a real `REJECT` row, keeps faith with the same M13 "no automatic decisions" discipline and this milestone's own explicit non-goals ‚Äî the mechanism is proven; no new enforcement is actually authorized against real data.
**Impact:** One new file (`backend/services/asset_definitions/enforcement_gate.py`), one new test file (`backend/tests/test_asset_registry_enforcement.py`, 48 tests), one modified file (`backend/services/asset_registry.py` ‚Äî new import, one new keyword-only parameter on `mint()`, one new call + structured log line + conditional raise, nothing else changed; `git status` confirms exactly this). No file from M10-M13 was touched; `enforcement_decisions.py`'s real table is unmodified (still zero `REJECT` rows). Regression-checked: the new suite (48/48) together with the full pre-existing `asset_definitions`/consultation/registry surface (`test_asset_registry.py`, `test_asset_registry_runtime_consultation.py`, `test_asset_definitions_enforcement.py`, `test_asset_definitions_conformance.py`, `_capability_view.py`, `_registry.py`, `_binding_resolver.py`, `test_ledger_validator_runtime_consultation.py`): 97/97 unchanged. Also ran the actual production call-path suites into `mint()` (`test_registry_service.py`, `test_registry_bootstrap.py`, `test_migration_executor.py`): 35/35 unchanged. Ledger Validator, Search, Optimizer, and Portfolio Engine were not touched, per scope. Default behavior (`enforcement_mode` unset, `ASSET_MINT_ENFORCEMENT_MODE` unset) remains byte-identical to pre-M14 `mint()` aside from one additional `INFO`-level log line per call. This completes the first Runtime Enforcement Stage R2 gate: narrow (one consumer), reversible (stateless, mode-per-call), explicitly scoped (one decision table, zero real `REJECT` rows), and observable (structured outcome fields + log line) ‚Äî exactly as the brief's Success Criteria require. Whether to author a real `REJECT` decision for any `AssetType` remains a separate, future, human-scoped decision.

## M23 ‚Äî Vocabulary Extension Stage: Bond Lifecycle Semantics

## M31.1 - Execution Instrument Facts Foundation

**Date:** 2026-07-14
**Problem:** Execution-facing code has no typed Registry-backed contract for describing an instrument. Existing execution taxonomy is locally inferred from DR-shaped symbols, ETF ticker allow-lists, caret-prefixed index symbols, and an EQUITY fallback. M31.1 is foundation only: establish the authoritative facts contract and resolver without migrating a consumer or changing any execution, optimizer, accounting, fee, funding, persistence, or frontend behavior.

**Decision:**
- Added `backend/services/execution_instrument_facts.py` with immutable `ExecutionInstrumentFacts`, `ExecutionFactProvenance`, and closed enums for the four resolution outcomes (`RESOLVED`, `UNKNOWN`, `AMBIGUOUS`, `NOT_TRADABLE`), five instrument forms (`EQUITY`, `ETF`, `DEPOSITARY_RECEIPT`, `OTHER`, `UNKNOWN`), and execution roles (`TRADABLE`, `REFERENCE`, `UNKNOWN`). `INDEX` is deliberately absent from instrument forms.
- The adapter delegates symbol/asset-id resolution to `services.registry_lookup.resolve_asset()` and relationship reads to `services.registry_service.get_relationships()`. It contains no identifier-matching implementation and does not query Registry ORM tables directly.
- The exact authoritative field for EQUITY/ETF is `models.asset.Asset.asset_type`, projected by `registry_lookup.AssetView.asset_type`. `AssetType.EQUITY` maps to execution form `EQUITY`; `AssetType.ETF` maps to `ETF`; all other resolved Registry asset types map to `OTHER`. There is no fallback to EQUITY.
- DR form is derived only from an outgoing `models.asset.AssetRelationship.relationship_type == DEPOSITARY_RECEIPT_OF` relationship. The relationship target is exposed as `underlying_asset_id`. An incoming relationship does not classify the underlying as a DR, and multiple outgoing DR-underlying relationships produce `AMBIGUOUS` rather than a guess.
- Market-index reference status is derived from the conjunction of authoritative Registry facts `Asset.asset_type == OTHER` and the current `AssetClassification` on dimension `ASSET_CLASS` with value `INDEX`; its Registry classification `source` is retained in execution-fact provenance. Such an asset has execution role `REFERENCE`, outcome `NOT_TRADABLE`, and form `OTHER`. Requiring `OTHER` prevents an ETF whose classification describes index exposure from being mistaken for the index reference itself. This is Registry classification consumption, not symbol-shape inference.
- Tradability and execution capabilities come from persisted Registry fields `Asset.tradable`, `Asset.fractional_support`, `Asset.lot_size`, and `Asset.settlement_cycle`. `registry_lookup.AssetView` was extended additively to project those fields plus classification provenance. `registry_lookup.Unresolved` now retains the typed identity-resolver verdict so execution can distinguish `AMBIGUOUS`/`CONFLICT` from `UNKNOWN` without parsing reason strings.
- Missing or malformed required Registry metadata produces an explicit `UNKNOWN` facts value with `UNKNOWN` form and role. Identity `CONFLICT` is conservatively represented as execution `AMBIGUOUS`; no candidate is selected.
- No eligibility enforcement was added. No existing consumer imports the new module, and `execution_penalty.py`, optimizer scoring, broker fees, funding/planning arithmetic, persistence models, APIs, and frontend code are unchanged. Consumer migration remains M31.2.

**Reasoning:** Asset Registry owns identity, structural asset type, relationships, classifications, and tradability. Execution owns a projection suitable for later judgment and planning, not a second classifier. Extending the existing immutable Registry read model preserves the established identity resolver's ambiguity and precedence rules while keeping the new adapter free of duplicate lookup logic. Explicit unresolved values ensure incomplete Registry coverage remains visible rather than being converted into a plausible but unsupported execution classification.

**Impact:** Added `backend/services/execution_instrument_facts.py` and `backend/tests/test_execution_instrument_facts.py`; additively extended `backend/services/registry_lookup.py`. Fifteen new test cases cover the contract, Registry-backed equity and ETF, separation of an index-tracking ETF from an index reference, DR relationship direction and underlying identity, unknown and ambiguous resolution, non-tradable index reference, incomplete metadata, capability projection, immutability, and behavioral/structural proof that regex, ticker allow-list, caret-prefix, and fallback classification are absent. Focused M31.1 plus Registry lookup tests: 33 passed. Relevant Registry and Registry-consumer backend regression suite: 188 passed. Existing execution/optimizer regression files: 53 passed and four pre-existing `test_optimizer_pipeline.py` failures caused by its stale two-argument `_consensus_engine(l2, l3)` calls after the production function gained a required leading argument; this exact failure set predates M31.1 and no optimizer file was changed. ## M31.2 - Execution Risk Consumer Migration

**Date:** 2026-07-14
**Problem:** `services/optimizer/execution_penalty.py` still owns a second, symbol-derived execution taxonomy after M31.1 established the Registry-backed `ExecutionInstrumentFacts` contract. Migrating the consumer must not turn incomplete Registry coverage into optimizer failures or unintentionally alter existing score, warning, slippage, or cap behavior.

**Decision:**
- `classify_execution()` and `compute_portfolio_execution_context()` now accept immutable `ExecutionInstrumentFacts`; neither function accepts a database session, resolves identity, queries ORM models, or imports Definition Runtime. Registry `RESOLVED`/`NOT_TRADABLE` facts select the scoring profile from `instrument_form`. EQUITY, ETF, and DEPOSITARY_RECEIPT map to the existing EQUITY, ETF, and DR judgment assumptions, preserving their existing liquidity, spread, risk, warning, slippage, penalty, and cap calculations.
- The `/analyze/optimizer` orchestration boundary already owns the SQLAlchemy session and complete `scores_map`, so it now calls `resolve_execution_instruments(db, tuple(scores_map))` once before pure scoring. The new batch adapter deduplicates queries and delegates identity to the existing `registry_lookup.resolve_many()` batch read rather than introducing another identifier lookup implementation. Adaptation failures are isolated per query; a batch read failure yields explicit `UNKNOWN` facts for the supplied set and never raises out of the optimizer path.
- The old DR regex, ETF ticker allow-list, caret-prefix index rule, and default-EQUITY projection were removed from `execution_penalty.py`. They remain only in the explicitly named `execution_penalty_compat.py`, whose module contract marks it as deletable, non-authoritative compatibility behavior. It is consulted as a shadow for resolved facts and controls scoring only for `UNKNOWN`/`AMBIGUOUS` facts (or legacy direct calls that supply no facts). It never creates, enriches, or backfills Registry facts.
- Registry facts are authoritative on shadow disagreement. Per-symbol context additively surfaces `resolution_status`, `instrument_form`, `execution_role`, `classification_source`, `classification_warning`, `legacy_asset_type`, and `classification_agrees`. Unresolved scoring identifies `LEGACY_COMPATIBILITY_FALLBACK` provenance and an explicit warning; resolved disagreements identify `SHADOW_MISMATCH`. These diagnostics are separate from `execution_warnings`, so agreed resolved instruments retain byte-equivalent risk warnings.
- A Registry market-index reference remains `NOT_TRADABLE` / `REFERENCE` / `OTHER` and receives a neutral descriptive `OTHER` risk profile; it is never classified as executable INDEX or EQUITY by the authoritative path. No optimizer or transaction blocking gate was added. Existing score-penalty and position-cap behavior remains in place unchanged, as required for this compatibility-first milestone.
- Existing API keys remain present. The additive diagnostic fields require no frontend behavior change; the frontend only renders `execution_risk` and `execution_warnings`, while its `asset_type` declarations were confirmed as the only frontend dependency on the legacy strings. No frontend file was modified.

**Deviations:** The Registry's existing `resolve_many()` API internally reuses the established per-query resolver and TTL cache rather than issuing one bespoke multi-symbol SQL statement. M31.2 deliberately reuses that authoritative read path: the optimizer performs one deduplicated batch request per run and contains no per-symbol Registry call, while Registry query optimization remains a Registry concern rather than duplicated execution-domain lookup logic. The compatibility classifier is retained in a separate module because removing it outright would change observable results for unresolved symbols before Registry coverage is complete.

**Reasoning:** The execution facts adapter owns descriptive identity projection; `execution_penalty.py` owns judgment. Passing facts into pure helpers keeps ORM lifetime and failure handling at the existing request boundary, makes batch behavior testable, and prevents Definition Runtime or Registry persistence concerns from entering scoring. Isolating legacy rules makes compatibility explicit and removable without presenting inferred values as authoritative facts.

**Impact:** Modified `backend/main.py`, `backend/services/execution_instrument_facts.py`, and `backend/services/optimizer/execution_penalty.py`; added the isolated `backend/services/optimizer/execution_penalty_compat.py` and `backend/tests/test_execution_penalty_m31_2.py`. ## M33.1 - Execution Intent Snapshot and Lifecycle Foundation

## M33.2 - Pure Execution Intent Contract and Transition Validator

**Date:** 2026-07-16

**Decision:** `ExecutionIntentSnapshot.content_hash` covers only what a human
reviewed: `terms_schema_version`, `intent_kind`, `terms` (allocations
canonically sorted by `(symbol, side)`), `workspace_id`, `portfolio_id`,
`effective_at`, `expires_at`, and `source_provenance` (canonically sorted).
It deliberately excludes `snapshot_id`, `intent_id`, `revision`,
`supersedes_snapshot_id`, `created_by_actor`, and `recorded_at` as
lineage/bookkeeping metadata rather than reviewed content. Ledger-derived
fulfillment (`assess_fulfillment`) is coverage-based (symbol/side presence),
not quantity-based, because M33 terms carry a target weight/value, not an
admission-ready quantity.

**Reasoning:** M33.1 section 8.4 specified that `content_hash` is "a
deterministic hash of canonical immutable content" without enumerating the
exact field set. Excluding identity/lineage/actor/recorded-at fields lets a
resumed-and-re-approved snapshot with unchanged terms keep the same content
hash across revisions, while approval binding still uses the exact
`snapshot_id` in addition to the hash (M33.1 ¬ß8.13), so this does not weaken
approval-hash binding. Quantity-aware fulfillment reconciliation was
explicitly deferred rather than guessed, since M33 terms do not carry an
executable quantity.

**Impact:** `backend/services/execution_intent_contracts.py` and
`backend/services/execution_intent_transitions.py` implement pure,
ORM-free contracts and a pure lifecycle transition validator only; see
`docs/implementation/M33_2_pure_execution_intent_contracts.md`. No model,
migration, endpoint, writer, or runtime adoption is introduced; M32 remains
closed and canonical execution planning remains NO-GO. A future adapter
milestone (M33.3) should study which legacy `RecommendationSnapshot`/
`UserExecutionDecision` rows can honestly produce a complete intent snapshot
before any persistence milestone is proposed.

## M33.4 - Historical Authority Certification and Reconfirmation Contract

**Date:** 2026-07-17

**Decision:** Historical approval may be recreated only from a trusted,
unrevoked, versioned authority certificate that proves the exact reviewed and
approved payloads, historical human identity and authority, immutable scope,
unambiguous lineage, UTC event time, and a binding to the predetermined M33.2
snapshot id and content hash. Lesser but non-conflicting evidence may seed only
a visibly non-authoritative pre-snapshot proposal requiring fresh human review;
conflicting and unusable cases remain quarantined, while rejected and expired
decisions remain out of scope.

**Reasoning:** M33.3 proved that persisted legacy rows alone have zero safely
exact-adaptable cases. Terms similarity, recommendation defaults, shadows,
transaction linkage, latest-row ordering, and operator assertions cannot fill
the missing historical actor or approval binding. Requiring a trusted
certificate to bind both historical evidence and the exact target snapshot
preserves M33.2's approval invariant; defining proposals as pre-snapshot data
prevents reconstructed terms from inheriting historical approval silently.

**Impact:** M33.4 is design-only and changes no M33.2 contract or runtime
behavior. No code, model, migration, repository, API, writer, persistence,
legacy conversion, or production import is added. M32 remains closed and
canonical execution planning remains NO-GO. The next bounded milestone is a
pure, ORM-free authority verifier and reconfirmation contract with fixtures,
not a legacy adapter or persistence rollout.

## M33.6 - Authority Evidence Availability and Issuer Governance Study

**Date:** 2026-07-17

**Decision:** Existing historical recommendation and decision evidence cannot
reach `CERTIFIED_EXACT` or `CERTIFIED_PROPOSAL_ONLY`. Historical records may at
most seed an explicitly policy-enabled, warning-rich `UNVERIFIABLE` proposal
that receives fresh human reconfirmation; rejected and expired decisions
remain out of scope, and conflicting cases remain non-proposal. For prospective
decisions, prefer direct fresh M33.2 approval of a predetermined frozen
snapshot id and content hash. Do not introduce certificate infrastructure
unless a later named relying party or compliance requirement justifies a
separate trust domain. Persistence readiness is
`PROSPECTIVE_CAPTURE_DESIGN_REQUIRED`.

**Reasoning:** Repository and configured-development-database inspection found
77 legacy decisions, including 21 `APPROVED`, but zero decision-owned
allocation payloads; no persisted actor; no actor-scoped authorization;
no frozen displayed/accepted payload digest; naive timestamps; no request or
correlation identity; and no audit, archive, certificate, trust, or revocation
store. The frontend submits only a decision label/notes, authentication reduces
the JWT to boolean validity, and backups/sync/shadows/transactions cannot fill
the missing authority. Signing those mutable facts later would authenticate a
new statement, not prove the historical human act. M33.2 already supplies the
needed direct approval invariant for new decisions, so certificates add value
only for an independently justified relying system.

**Impact:** M33 does not proceed to authority or reconfirmation persistence.
The next bounded milestone is a design-only prospective human-identity,
authorization, frozen-display, direct-approval, correlation, and atomicity
contract. No code, model, migration, endpoint, frontend, writer, signing/key
operation, adapter, legacy conversion, production behavior, or Graphify output
changes. M32 remains closed and canonical execution planning remains NO-GO.
